项目难点分析
1. NL2SQL对话式数据查询（核心难点）
销售人员用自然语言描述数据需求，LangChain编排NL2SQL链路：用户输入 -> 意图识别 -> Schema元数据注入（Oracle表结构/字段中文注释） -> 大模型生成SQL -> SQL安全校验 -> 执行查询 -> 结果格式化+图表渲染
难点一：Oracle SQL方言适配，分页查询用ROWNUM/FETCH FIRST而非LIMIT，日期函数用TO_DATE/TRUNC，字符串处理差异等，需在Prompt中注入Oracle特有语法示例（Few-shot）
难点二：多表复杂关联，CRM涉及客户表、商机表、合同表、跟进表、区域表等多表JOIN，大模型需要理解表间关系才能生成正确SQL
难点三：数据权限控制，普通销售只能查自己的数据、销售经理能查团队数据、区域总监能查整个区域数据，生成的SQL需要动态注入权限过滤条件
2. 基于机器学习的销售额预测模型
利用历史3年的成交数据，提取特征：商机阶段分布、客户行业占比、历史同期销售额、销售团队人数、线索转化率、平均成交周期等
使用XGBoost训练回归模型，通过时间序列交叉验证评估模型效果，输出未来季度的预测值与置信区间
难点在于数据稀疏性（部分区域/行业样本不足）、季节性波动的特征捕获、以及模型的持续迭代更新（新数据增量训练），需要设计定时任务（XXL-JOB）周期性重训模型
3. AI商机赢率分析与智能跟进建议
对每个商机，基于其阶段、金额、客户等级、竞对情况、跟进频率、停滞天数等特征，利用分类模型（随机森林）预测赢单概率
利用大模型分析商机的所有跟进记录，自动总结当前进展、识别风险点（如"客户提到预算紧张"、"竞对报价更低"），并生成针对性跟进建议
难点在于跟进记录的文本质量参差不齐（有的只写了"电话沟通"），需要大模型在信息不足时合理推断而非编造
4. 客户公海池与智能分配机制
公海池规则引擎：超过N天未跟进的客户自动回收至公海、同一销售持有客户数上限控制、新线索按区域/行业/销售历史成交率智能分配
难点在于分配算法的公平性与效率平衡（既要能力匹配又要负载均衡），以及并发抢客户时的分布式锁控制（Redis分布式锁）
5. 多层级数据看板的实时性与钻取
四级看板（全国 -> 区域 -> BU -> 个人）支持逐级钻取，点击某区域可下钻到该区域的BU数据，再下钻到具体销售
难点在于Oracle下大数据量聚合查询的性能优化（使用物化视图预聚合 + Redis缓存热数据），以及看板数据的准实时更新策略（定时刷新 vs 增量计算）
6. 销售漏斗阶段管理的灵活性
商机阶段漏斗支持自定义配置（不同BU可能有不同的阶段定义），阶段推进时触发校验规则（如"方案报价"阶段必须上传报价单）
难点在于漏斗阶段的动态配置不能影响历史数据的统计口径，以及阶段回退场景的业务逻辑处理
7. 跟进记录的AI摘要与知识沉淀
销售的跟进记录长期积累后形成企业客户知识资产，利用大模型对跟进历史进行摘要，生成客户的"关系图谱"（关键决策人、偏好、痛点、成交路径）
新销售接手客户时，AI自动生成"客户交接简报"，快速了解客户历史，降低客户流失风险
8. EasyExcel大批量数据导入导出
线索批量导入（万级数据量）、销售报表多Sheet导出，需要使用EasyExcel的SAX模式避免OOM
难点在于导入时的数据校验（手机号/邮箱格式、客户查重、区域编码映射），以及异步导入的进度反馈（WebSocket推送导入进度）
